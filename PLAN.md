# План завершения backend проекта PIONEER

## 1. Краткий анализ текущего состояния проекта

- Текущий backend уже не является "пустым MVP", а представляет собой частично реализованный Django/DRF-монолит с доменными apps: `users`, `organizations`, `services`, `bookings`.
- Источником правды сейчас должен считаться код: `README.md`, `API_DOCUMENTATION.md`, `DATABASE_SCHEMA.md` и `DOCKER_SETUP.md` уже выровнены под текущую реализацию, но сиды и часть тестовых скриптов всё ещё местами описывают старую телефонную/SMS-модель и устаревшие контракты.
- Авторизация в коде уже переведена на email-коды + JWT + `UserSession`, а ТЗ ориентировано на мобильное приложение и вход по телефону. Для backend нужно закрепить именно web + email-only вариант.
- Базовые CRUD и role-based queryset/permissions уже есть, но бизнес-логика бронирований пока не доведена до полностью рабочего состояния: нет доступных временных интервалов, нет проверки конфликтов, нет полноценной записи `BookingItem` из API, нет управляемых переходов статусов.
- В коде и вспомогательных скриптах всё ещё видны следы старой модели: `seed_demo.py` с phone/password, часть тестовых/ручных сценариев и не до конца стабилизированный auth surface рядом с universal endpoints.
- Production-ready состояние пока не достигнуто: один файл настроек, `DEBUG=True`, `ALLOWED_HOSTS=[]`, `CORS_ALLOW_ALL_ORIGINS=True`, секреты в `.env.docker`, Docker стартует через `runserver`, нет Nginx/HTTPS/domain setup.
- Полноценную runtime-проверку в момент анализа подтвердить не удалось: Docker daemon был недоступен, а запуск локального Python вне workspace ограничивался песочницей. Поэтому план ниже построен по статическому анализу ТЗ, README и текущего кода.

## 2. Что уже реализовано

- Кастомная модель `User` с UUID, email-first авторизацией, ролями `ADMIN` / `ORGANIZATION` / `CLIENT`, полем принятия privacy policy и хранением текущей активной сессии.
- Модель `UserSession` и базовая логика "одна активная сессия" при логине.
- Email auth flow: регистрация, логин, recovery, logout, выдача access/refresh JWT, verify endpoint, коды с TTL/лимитом попыток/cooldown/dev-code; после последних обновлений уже добавлены универсальные публичные endpoints `/api/users/auth/send-code/` и `/api/users/auth/verify-code/`.
- Кастомный admin login через email-код.
- Доменные модели и CRUD для `City`, `Organization`, `Service`, `ServiceItem`, `Booking`, `BookingItem`.
- Базовая role-based фильтрация queryset и object-level permissions для организаций, услуг и бронирований.
- Базовые сериализаторы, роутеры, admin-конфигурации, seed-команды, Dockerfile и `docker-compose.yml`.
- Кастомный exception handler и JWT middleware.
- Часть автотестов для `users`; в корне есть ручные smoke/security-скрипты.

## 3. Что реализовано частично

- Документация уже выровнена под текущие пути, payload-ы и терминологию; дальнейшей ревизии требуют прежде всего сиды, ручные скрипты и тестовые заготовки.
- Авторизация: email-коды и новые универсальные `/api/users/auth/send-code/` + `/api/users/auth/verify-code/` уже реализованы, но surface пока смешанный: рядом живут legacy `/auth/email/...`, в auth-views есть дубли/неоднозначности, а logout/refresh/session invalidation не доведены до строгой серверной консистентности.
- Роли и права: базовая схема есть, но не везде зафиксированы ограничения на запись и допустимые действия по статусам.
- Бронирования: CRUD есть, но нет полноценной доменной логики записи на слот и обработки заявки организацией.
- Элементы услуг и элементы бронирования: модели есть, но API-поток создания/валидации/заморозки цены не завершён.
- Тесты: покрыт в основном auth-слой; для `organizations`, `services`, `bookings` реальных API-тестов почти нет.
- Docker/dev setup: для локального запуска годится, для production нет.
- Сиды и вспомогательные скрипты: часть актуальна, часть осталась от старой phone/password-логики.

## 4. Что еще нужно реализовать

- Адаптировать ТЗ под web-приложение и закрепить backend-истину: без mobile/Flutter/Android/SMS.
- Стабилизировать auth-логику после появления универсальных `/api/users/auth/send-code/` и `/api/users/auth/verify-code/`: убрать дубли и неоднозначности, синхронизировать routes, views и документацию, зафиксировать канонический auth surface.
- Доделать production/dev settings и конфигурацию окружений.
- Закрыть расхождения по моделям, сериализаторам, permissions и API-контрактам.
- Зафиксировать в backend обязательный адрес организации: город + адресная строка, при необходимости координаты (`latitude` / `longitude`) для дальнейших интеграций.
- Добавить отдельную backend-сущность машин текущего пользователя (`ClientVehicle`) в `users`: один user -> много машин, CRUD только по своим машинам, минимальные поля `brand`, `plate_number`, `tire_diameter`.
- Добавить сущности параметров услуги и их опций, чтобы клиент выбирал не только услугу, но и конкретные параметры услуги по ТЗ.
- Добавить backend-поток подключения организации через заявку, как в ТЗ, но в web-варианте.
- Доделать booking flow целиком: выбор услуги, выбор параметров услуги, выбор организации, выбор даты и времени, выбор машины пользователя, сохранение заявки, последующую обработку заявки организацией.
- Синхронизировать ручную документацию и/или ввести автогенерируемую API-схему.
- Усилить безопасность, логирование, обработку ошибок и готовность к продакшену.
- Подготовить реальный production deploy на Ubuntu/VPS с Docker Compose, Nginx, HTTPS и доменом.

### Ключевые backend-изменения в публичных интерфейсах

- Канонический публичный auth surface для web-клиента: базовые endpoints `/api/users/auth/send-code/` и `/api/users/auth/verify-code/` уже есть и должны быть доведены до стабильного состояния; рядом остаются `logout`, `jwt/verify`, отдельный `token/refresh` и при необходимости recovery flow. Legacy `/api/users/auth/email/...` нужно либо оставить как compatibility layer, либо убрать из публичной документации после синхронизации routes/views/docs.
- Канонические роли в API и БД: `ADMIN`, `ORGANIZATION`, `CLIENT`; `OWNER` оставить только как описание в текстах, не как API-константу.
- Организация в backend обязана иметь адресный блок: `city`, `address`, при необходимости `latitude` и `longitude`; эти поля должны быть отражены в модели, API, сериализаторах и тестах.
- Новый backend-блок машин пользователя: модель `ClientVehicle` размещается в `users`, связь `один user -> много машин`, API ограничен CRUD только своих машин, минимальные поля `user`, `brand`, `plate_number`, `tire_diameter`, `created_at`, `updated_at`.
- Новый backend-блок параметров услуги: `ServiceParameter` и `ServiceParameterOption` в `services`, сохранение выбранных параметров в бронировании через отдельную сущность выбора.
- Новый backend-блок для подключения организаций: web API для создания и отслеживания заявки + admin/API-модерация.
- Новый backend-блок для слотов: endpoint доступных интервалов по услуге и дате.
- Новый набор action-endpoints для бронирований: подтверждение, отмена, завершение, отклонение по допустимым ролям.
- `Booking` получает связь с выбранной машиной пользователя через `client_vehicle`; если услуга зависит от параметров колеса/шины, backend валидирует совместимость выбранной услуги/опций с `tire_diameter` машины.
- Новые/уточнённые модели: `OrganizationConnectionRequest`, `ClientVehicle`, `ServiceAvailabilityRule`, `ServiceParameter`, `ServiceParameterOption`, `BookingParameterSelection`, опционально `AccountDeletionRequest`; `BookingItem` creation flow и связь `Booking -> client_vehicle` становятся частью публичного booking API.

## 6. Ключевые тестовые сценарии

- Универсальный auth flow через `/api/users/auth/send-code/` и `/api/users/auth/verify-code/`: регистрация нового клиента, логин существующего пользователя и завершение незавершённой регистрации.
- Логин существующего пользователя, refresh токена, logout и повторный запрос с деактивированной сессией.
- Синхронизация канонических auth routes/views/docs: legacy `/api/users/auth/email/...` не расходятся по смыслу с публичным контрактом или исключены из него.
- Ограничение повторных отправок кодов, лимит попыток, блокировка и поведение в prod без `4444`.
- Создание и обновление организации с обязательным адресным блоком: `city`, `address`, опционально `latitude` / `longitude`.
- Создание/модерация заявки на подключение организации и доступ к статусу со стороны партнёра.
- CRUD машин текущего пользователя (`ClientVehicle`): создание, список своих машин, редактирование и удаление.
- Проверка связи бронирования с конкретным `ClientVehicle` и отказ, если машина не принадлежит текущему пользователю.
- Получение каталога параметров услуги и допустимых опций параметров для выбранной услуги.
- Полный booking flow: выбор услуги, выбор параметров услуги, выбор организации, выбор даты и времени, выбор машины пользователя, сохранение заявки и последующая обработка заявки организацией.
- Получение слотов по услуге и отказ в двойном бронировании.
- Создание бронирования с `BookingItem`, выбранными параметрами услуги, выбранной машиной пользователя, корректной ценой, валидным статусом и проверкой совместимости по `tire_diameter`, если услуга зависит от параметров колеса/шины.
- Подтверждение, отклонение, завершение и отмена бронирования по допустимым ролям.
- Импорт prod settings, health endpoints, генерация API schema, миграции и сиды.
- Smoke после деплоя по домену: auth, docs, admin login, booking flow.

## 7. Допущения и выбранные решения

- Целевой прод: Ubuntu/VPS + Docker Compose + Nginx + HTTPS + домен.
- Канонический backend-вариант: web application + email-коды; mobile, SMS, Flutter и Google Play из ТЗ исключаются.
- Канонический публичный auth surface для web-клиента: `/api/users/auth/send-code/` и `/api/users/auth/verify-code/`; legacy `/api/users/auth/email/...` допускаются только как transitional compatibility layer до полной синхронизации routes, views и документации.
- Из ТЗ остаются в scope: роли, организации, адреса организаций, каталог ТС клиента, параметры услуги, бронирование, статусы, one-device policy, privacy/legal минимум, подключение организаций, PostgreSQL, учёт таймзон.
- `ClientVehicle` живёт в `users` app; один пользователь может иметь много машин; минимальный backend-состав полей: `user`, `brand`, `plate_number`, `tire_diameter`, `created_at`, `updated_at`.
- `Booking` хранит ссылку на выбранную машину пользователя через `client_vehicle`; если услуга зависит от параметров колеса/шины, backend валидирует совместимость выбранной услуги/опций с `tire_diameter` машины.
- GooglePay, 1C:CRM, push, карты, соцсети и mobile-only UX не входят в план завершения backend MVP.
- Для слотов в MVP принимается вместимость `1` на услугу, пока в домене нет отдельной модели постов/боксов/ресурсов.
- README и API docs должны быть приведены к коду, а не наоборот.
